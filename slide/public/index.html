背景 / Two paradigms of AI

符号主义输了五十年——但它留下的工具，正是这一代赢家在用的工具

上一代：符号主义 AI（1956 – 1990s）

  • 知识表示、形式逻辑、规则推理、搜索
  • Lisp / Prolog / 专家系统 / Cyc / SOAR
  • 口号："AI = 写下世界的规则"
  • 败因：感知做不动、知识工程不可扩展

这一代：连接主义 AI（2012 – 今）

  • 梯度学习、表示学习、Transformer
  • CNN / RNN / Deep Learning / LLM
  • 口号："AI = 喂数据，让权重涌现"
  • 胜势：感知、对话、代码生成

符号主义输了战争，却留下了软件工程最深刻的遗产——REPL、Lisp、形式语言、求值器、homoiconic 文化。它没死，它沉到了我们每天用的开发工具底层。

30 秒过：50–90 年代符号 AI 主导，专家系统幻灭后让位给连接主义。但 LISP/REPL/eval 这些工具范式渗入了 Unix、Emacs、Smalltalk、Ruby、Python、JavaScript——它们是隐形赢家。这页是为后面"两条线汇合"埋伏笔。

一、Tool Calling 已经走到了十字路口

四个阶段：抽象单位在不断上移

阶段 抽象单位 谁决定能力边界 典型问题
Function Calling 单个 endpoint 工程师 每个能力都要人类预先注册
MCP 一组 endpoint 工程师 + 协议维护者 供给侧解耦了，抽象层次没动
CLI Agent shell 表达式 操作系统 把“调用”偷换成“求值”——但解释器受限
REPL 宿主语言表达式 Agent 自己 能力上限 = 这门语言可表达的一切
这张表是全场骨架。前三阶段都在"调用工具"，REPL 是第一次让模型直接面对一个图灵完备的求值环境。

CLI 的偷渡

CLI 的威力不是工具更多，而是把“调用”悄悄换成了“求值”

表面上

  • Agent 学会用 bash，所有 Unix 工具瞬间可用。
  • 看起来只是“工具表更大”。

实际上

  • 执行单位从“注册过的函数”变成“图灵完备解释器求值”。
  • 能力上限从“工程师预想过的任务”变成“这门语言能表达的一切”。
CLI 指对了方向，但解释器太弱——bash 的类型是字符串、组合是 | 和 &&、状态是全局环境变量。
关键过渡：CLI 已经走对了一半，REPL 是把这半步走完——给 Agent 一个真正的语言。

二、MCP 正在重新发明一个半成品的解释器

看一眼复杂的 MCP server 实际在做什么

变量绑定

“把上一个查询结果传给下一个工具”——在重新实现 let 和作用域。

条件分发

“如果场景里有 X 就做 A”——在重新实现 if。

批处理 / 循环

“对列表里每一项调用工具”——在重新实现 for。

工具组合

“先查、再改、再验证”——在重新实现函数组合。

异步

等加载、等动画、等网络——在重新实现 coroutine / Promise。

结论

每一条都是在 JSON schema 上打补丁，模拟一门语言本来自带的能力。

这页可以现场点名：你们维护的任何一个 MCP server，大概率正在偷偷实现这五件事中的几件。

从护栏到天花板

当模型已经能稳定生成代码，schema 就从约束变成限制

过去：schema 是护栏

  • 模型生成代码不稳定，会幻觉路径与函数名。
  • JSON schema 用强类型约束消除幻觉。
  • MCP 在这个阶段是正确选择。

现在：schema 是天花板

  • 模型生成代码的能力已经 ≥ 调用工具的能力。
  • Agent 用自然语言操作一层翻译层操作另一层翻译层。
  • 工具面被枚举量级 cap 死，复合表达力被绕路。
“eval is evil” —— 直到模型写代码比 schema 还稳。
不是要否定 MCP，是说它的适配区在收缩。"eval is evil" 是 JS 社区二十年来的口号，这里恰好被翻转——它评价的是不可控的字符串，而模型现在写出来的是结构化、可读、可 diff 的真代码。

三、核心论断

Token is language. Language is evaluation. Evaluation is the universal tool.

模型已经在生成一门真实编程语言的 token 了——那就直接 eval 它。

不需要 {"tool": "load_asset", "params": {...}} 这层包装，再让 server 反序列化回 C# 去反射调用——直接把源码送进解释器求值就结束了。

下面三页：这一论断带来的三个结构性优势——Schema-free、组合性、语言级异步。

这是全场最重要的一句。讲完之后可以停一拍。

优势 1 / Schema-free

工具面不再需要被枚举

传统 tool-calling

  • 工具面封闭——注册 100 个，就只会用这 100 个。
  • 新需求需要工程师先回去加一个 endpoint。

REPL 范式

  • 工具面 = 整个宿主语言生态：所有库、所有类型、反射可达的一切。
  • 面对未见过的状态，Agent 可以即席组合标准库 + 用户代码 + LINQ + 反射。

这不是“工具多一点”的量变，是质变：从封闭、有限集合 → 开放、无穷表面。没有人需要提前为它注册一个 endpoint。

现场可以问："你们维护的 MCP server 有多少个 endpoint？面对一个没设计过的查询，agent 现在能做什么？"

优势 2 / 组合性

把能力塞进语言、而不是塞进协议

协议层做五件事

  • 变量、条件、循环、组合、错误处理
  • 每一件都是 schema 上的临时补丁
  • 跨工具拼装靠 prompt 而不是语法

语言做一件事

  • 立刻继承变量、闭包、泛型、管道、async/await
  • 组合性是语法、不是约定
  • 错误处理走语言原生的 try / Result
这就是 Greenspun 第十定律的直接后果。

“计算机科学里所有问题都可以用多加一层间接来解决——除了'间接太多'这个问题。”

—— David Wheeler / Butler Lampson。我们正在解决的恰好是后者。

Wheeler 这句话是工业界最爱引的金句之一。今天反过来用：MCP 是那"多出来的一层"，我们要的是干掉它。

优势 3 / 语言级异步

这是被严重低估的一点：异步从分布式问题降维成语言表达

RPC 架构里

  • 方案 A：让 agent 轮询——浪费 token、破坏控制流。
  • 方案 B：server 侧自建状态机——每个新场景都得重做。
  • MCP 请求/响应语义是一次性的，没有“你帮我 tick 一下”这种 idiom。
// REPL 里就是语言原生控制流 IEnumerator ComplexSetup() { EditorSceneManager.OpenScene("…/TestScene.unity"); yield return null; // 等一帧 var go = new GameObject("TestEnemy"); yield return new WaitForSeconds(2f); // 真实等待 go.GetComponent<Health>().Damage(10); yield return "done"; // 最后值 → 响应 }

Agent 返回一个 coroutine / Promise，server 自己把它 drive 到完成，再把最终值写回。异步从“上层要解决的分布式问题”降维成“语言原生支持的一种表达”。

这页是 Unity REPL 章节的伏笔。讲到这里听众应该已经能预感到：游戏引擎里 coroutine 满天飞，正是 REPL 的舒适区。

优势 4 / Control plane = 观察面

REPL 不只是执行面——它跑在宿主进程内，所有内存状态都是可询问的

全状态访问

REPL 不跨协议边界。Unity 里整个 Editor API、所有 GameObject、组件内部字段、反射可达的一切；浏览器里 DOM + JS heap + globals + window；server 进程里全部 in-process state。

MCP 只能看见 server 作者愿意暴露的视图——内部状态机基本不可见。

形式化验证

求值过程里可以直接断言不变量、属性、规格——不是 println，是 verifiable predicate。Property-based 检查、状态机性质、可执行规格。

验证逻辑住在代码里，可 review、可 diff、可 reuse；不是工具表里的一行说明。

行为验证

一次表达式里完成：驱动系统进入特定状态 → 观察轨迹 → 检查迁移合法。不止"任务做完没"，而是"过程是否合法、状态机有没有走野路"。

失败时返回 typed diff 与反例，而不是模型自述"我觉得 OK"。

MCP 验证只能为每个 invariant 新增一个 tool，每次都付一次 round-trip 成本，而且观察会再次进 transcript。REPL 验证就是表达式的一部分——成功/失败/反例一并以 compact 形式返回。

Agent 从"产生输出"升级到"产生可被自己验证的输出"。这是 control plane 才有的特权。

这页是从安全章节抽出来的，但本质上是优势——verification 不只是 risk control，是 capability。它是 control plane 才能给的：能读内存、能跑 invariant、能 trace 状态机迁移。对于 Unity 这种"状态深、组件多"的环境尤其关键，因为外部能看见的就是渲染结果，所有真状态都在内存里。

四、Exploration → Crystallization

Agent 自己编织工具链：工具不再是输入，而是工作产物的一部分

Explore 陌生项目里像新工程师一样交互式探索：列类型、dump 状态、跑小片段。
Recognize 发现某套动作序列有价值，值得复用。
Crystallize 把它固化成项目里的脚本文件，下次一行调用复用。

取消了一个根深蒂固的分工——工具的作者和工具的使用者不再是两个角色。MCP 的工具生命周期在协议外，REPL 的工具生命周期在代码里：提交一个文件就完成了。

这页是文章第四节，也是后面 Unity 示例的核心叙事——agent 先 live session，再把成果落成 Editor 脚本。

五、终局

再往上没有更高的抽象层了——你已经直接面对图灵完备

当抽象单位 = 宿主语言本身时，任何进一步的优化只能发生在语言内部（更好的库、更好的 sandbox、更好的错误消息），而不是协议层的重新设计。

诚实地引一句反方

“当心 Turing tar-pit——在那里一切皆有可能，却没有一件事是容易的。”

—— Alan Perlis, Epigrams in Programming (1982)

所以重点不在"图灵完备"

重点在它是一门人类工程师已经用了几十年的语言：有库、有 IDE、有 stack trace、有 review 文化。Agent 站在这套生态上，而不是站在沼泽上。

Perlis 这句是反 Lisp/REPL ideology 最经典的警告，必须正面应答——我们不是给 agent 一个空白图灵机，而是给它一门成熟语言的全套工程基础设施。

Rethink / 两条线在 REPL 处汇合

连接主义模型学会了输出符号——接住它的，正是符号主义留下的求值器

连接主义 LLM 生成 C# / Python / SQL

token 即语言
REPL 两代 AI 的
工程接合面
符号主义 Lisp · 1958 起
eval-print loop

MCP 试图给连接主义模型再罩一个小型符号系统（JSON schema + 工具表）——可模型早就会用完整符号系统了。让神经网络去写代码、让 Lisp 的后人去执行代码，是这两代 AI 在工程上最自然的和解。

前一代的工具是这一代的执行层。Unity REPL 不是技术潮流，是一次迟到了 30 年的握手。

这页是上半场最深的一击。可以慢一点讲：1980 年代专家系统衰退时，符号 AI 的"输"并不是工具的输——LISP/REPL 这些范式渗透到了软件工程的底层。现在连接主义模型生成的最值钱的输出恰好是源码，那么承接它的当然是 REPL。这就是为什么 Unity REPL 比 Unity MCP "更对"——不是工程偏好，是范式必然。

Crystallization / 沉淀

探索值得复用时，Agent 把它自己提交到 Assets/Editor/

// 由 agent 自己生成并提交到版本控制 // Assets/Editor/TurretRangeProbe.cs public static class TurretRangeProbe { public static string RunAt(Vector3 pos) => string.Join("\n", GameObject.FindObjectsOfType<Turret>() .Select(t => $"{t.name}: {Vector3.Distance(t.transform.position, pos)}m")); } // 下次一行复用： // TurretRangeProbe.RunAt(new Vector3(10, 0, 5));

工具链不是 Unity 团队维护的，不是插件作者维护的——是这个 agent 在这个项目里锻造出来的。像任何工程师写的代码一样可被 review、复用、演进。

先消除一个误解 / Control plane vs tool table

真正的区别不是"能不能写代码"——而是 Action 和 Evaluation 发生在哪里

形态 Action 位置 Evaluation 位置 反馈周期
MCP + 写代码 Unity 外部 日志 / 测试 / 再次查询 edit → compile → run → observe（外部 batch loop）
Unity REPL Unity 内部（主线程） live runtime state eval → observe → adapt（内部 live loop）
MCP 是协议层抽象；Unity REPL 是执行层抽象。

常见反问："如果 MCP 加一个 execute_code 工具呢？"
那它的有效部分就已经变成 REPL——只是外面包了一层 MCP transport。对 Unity 这种复杂运行环境，最终需要的核心原语是 eval，而不是 endpoint table。

这页解开一个常见误解：不要把对比讲成"MCP 不能写代码"——MCP 当然能让 agent 写 .cs 文件、触发编译、跑测试。但默认工作流是外部 batch loop，反馈周期重。REPL 是内部 live loop，act + verify + adapt 都在同一个 Unity runtime 里。execute_code 的反问要直面：那等于承认 eval 才是核心原语。

Agentic Loop 实验 / open door → survive laser → extend bridge → reach exit

REPL 在运行时层：实时修改、查看、验证状态

左 · MCP + 写代码（外部 batch loop）

  • Round 1：写代码修门 → 编译 → 运行 → 激光仍杀 bot
  • Round 2：写代码修激光 → 编译 → 运行 → 桥仍缺失
  • 视频窗口结束：NO PASS

右 · Unity REPL（运行时层 live loop）

  • 直接改 live scene：开门、关激光、延伸桥
  • 立刻读 runtime state：门状态、HP、bot 位置
  • 跨帧等待后继续观察：yield / WaitForSeconds
  • 修改 → 查看 → 再修改 → 验证：PASS

细节展开 / 同一台 Chrome、同一个 chrome-devtools-mcp@latest

区别全在"谁在调用工具"——以及调用之间发生了什么

Pure Chrome MCP（baseline）

  • Codex 直接拿到 chrome-devtools-mcp 的工具表。
  • 每个浏览器原语 = 一次顶层 Codex action。
  • evaluate_script 反复出现——读 DOM、抓字段、点元素。
  • 有些调用看不见 Chrome 动作（只读 state）但依然消耗一整次 agent turn。
  • 页面观察每次都回到模型 transcript。

36 个 mcp_tool_call · median 0.9s · P90 5.1s

Interactive REPL（同样的 chrome-devtools，包装后注入）

  • mcp2repl 把同一个 server 的工具作为库函数注入 evaluator。
  • Codex 写小过程：navigate → extract → normalize → checkpoint。
  • Chrome 在 evaluator 内连续动作；模型只收到 compact checkpoint。
  • 中间 DOM、原文、临时结果不进 transcript。

5 个 command_execution · 所有 eval ≤3.0s

最大动作间隔 22.3 秒——这不是 Chrome 在等，是 Codex 在写下一段 compound procedure。慢的是模型推理，不是浏览器。所以 REPL 的真正收益是把"被观察淹没的模型推理"换成"直接产代码的模型推理"。

这页是讲给"我们已经在用 Chrome DevTools MCP"的工程师听的。关键洞察：mcp2repl 不是替代 chrome-devtools-mcp，而是把它再降一层——让它从 Codex 的工具表变成 evaluator 里的库。同一个底层 server，同一台 Chrome，同一组浏览器能力，但 agent 的 action 单位完全不同。

边界 / Risk surface

REPL 更强，所以边界要比 MCP 更明确

权限

把 eval 面限制在任务需要的能力内。Editor-only、PlayMode 隔离、不默认开放文件系统/网络/进程。

审计

记录 evaluator commands、loaded assemblies、artifact handles、外部副作用。

验证（回扣"优势 4"）

把形式化与行为验证当作执行结果的必返字段：scene hash、object counts、invariant violations、状态机迁移轨迹——而不是模型自述。

直接工具调用的风险是"误点一个按钮"；REPL 的风险是"执行一段程序"。能力上移时，sandbox、allowlist、timeouts、dry-run、validation 也要一起上移。

With great eval comes great responsibility. —— 改编自 Uncle Ben

这页要平衡——不把 REPL 讲成银弹。强执行面需要强约束。Uncle Ben 是给现场气氛松一下的——但这句话本身是认真的。

怎么落 / Decision tree

按任务复杂度选择抽象层，不要一刀切

Direct MCP

一次性动作、低观察量、几乎不需要循环或状态。

例：开页面、读标题、点击一个已知按钮。

Interactive REPL

需要探索、观察量大、多实体比较、可逐步验证。

例：购物研究、复杂表单、Unity 场景调试。

Crystallized program

流程稳定、输入变化小、验证规则明确、要重复运行。

例：批量审计、固定迁移、可复用 Editor 任务。

成熟曲线：Direct MCP → Interactive REPL → Crystallized program。Agent 自己沿曲线向前。

推论：premature crystallization is the root of all evil. —— 致敬 Knuth；探索没收敛就别急着固化。

"简单事保持简单，复杂事上 REPL，稳定事产品化。"过早 crystallize 等于过早优化——会把 agent 还没想清楚的探索路径冻成石头。

开始使用 / Unity REPL is open source

贴一段 prompt 给你的 agent，它自己装完、握手、汇报

Add "com.lambda-labs.unity-repl": "https://github.com/LambdaLabsHQ/unity-repl.git" to the dependencies in Packages/manifest.json. Then register the Unity REPL skill: npx skills add ./Packages/com.lambda-labs.unity-repl If that fails, the skill definition is at ./Packages/com.lambda-labs.unity-repl/.agents/skills/unity-repl/SKILL.md — use your agent runtime's skill installer to register it. Finally, verify the REPL server is working by evaluating Application.unityVersion through the skill.

收束 · Callback

一句话：MCP 标准化工具，REPL 标准化工具的组合方式

01

任何足够成熟的 tool-calling 都在偷偷拼凑半个 REPL——Greenspun 第十定律。

02

Token 已经是语言。Schema-free、组合性、语言级异步——三件事一起得到。

03

Unity REPL 把它落到了一个最难的引擎；agent 自己锻造工具链。

回扣开篇：上一代符号主义 AI 留下了 REPL；这一代连接主义 AI 学会了写代码——前者是后者的执行层。Unity REPL 是这两代 AI 在工程上迟到的握手。
